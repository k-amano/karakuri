# Xolvien — 改修要件

## 1. 実装方針

### 1.1 実装の優先順位

以下の順序で実装を進める。

1. テスト実行 + 自動修正ループ（単体テスト → 結合テスト → E2Eテストの順に段階実装）
2. PR自動作成
3. GitHub Issue連携（Webhook → タスク自動作成・実行）

マルチユーザー対応（ストリーミングのブロッキング解消・GitHub OAuth認証等）は、シングルユーザーでの全機能実装完了後に着手する。

---

## 2. 実行フロー設計

### 2.1 基本方針

- ステップごとにユーザーの確認を挟む（意図と異なる方向への進行を早期に防ぐ）
- コミットは各ステップ完了時に都度行う（全完了まで待たない）
- pushは最終確認後に一括で行う
- テストコードはClaudeが自動生成・修正する（ユーザーがテストコマンドを入力する必要はない）
- テストコマンドはClaudeがリポジトリの構成（package.json、pyproject.toml等）から自動判断する

### 2.2 新規作成のフロー

```
1. 指示入力
2. 要件確認（Claude ↔ ユーザー）
3. プロンプト確認 → ユーザーが承認
4. Claude が実装コード + テストケース一覧を同時生成
5. ユーザーがテストケースを確認 → 承認 or 修正依頼
6. Claude がテストコードを生成・単体テスト実行
7. 失敗なら自動修正ループ（最大3回）
8. ユーザーが実装を確認（変更ファイル一覧 + 必要なら diff を別画面で表示）
   → 承認 → 自動コミット（実装 + テストコード）
   → 差し戻し → ステップ3に戻り指示を修正（前の実装は履歴として残る）
9. テスト結果をMarkdown形式でファイル保存
10. 結合テスト・E2Eテストへ（フェーズ2以降）
```

### 2.3 既存コード修正のフロー

```
1. 指示入力
2. 要件確認（Claude ↔ ユーザー）
3. プロンプト確認 → ユーザーが承認
4. Claude が修正コード + 既存テストケースの差分・追加テストケースを同時生成
5. ユーザーがテストケース（追加・変更分）を確認 → 承認 or 修正依頼
6. Claude がテストコードを修正・追加し、単体テスト実行（既存 + 新規）
7. 失敗なら自動修正ループ（最大3回）
   ※ 既存テストも含めて実行し、デグレード（既存機能の破壊）も検出する
8. ユーザーが実装を確認（変更ファイル一覧 + 必要なら diff を別画面で表示）
   → 承認 → 自動コミット
   → 差し戻し → ステップ3に戻り指示を修正（前の実装は履歴として残る）
9. テスト結果をMarkdown形式でファイル保存
10. 結合テスト・E2Eテストへ（フェーズ2以降）
```

### 2.4 ユーザーの確認ポイント

| タイミング | 確認内容 | 承認後の動作 | 差し戻し時の動作 |
|---|---|---|---|
| ステップ3 | プロンプトが意図通りか | 実装 + テストケース生成開始 | 指示を修正して再生成 |
| ステップ5 | テストケースが網羅的か・意図通りか | テストコード生成・実行 | テストケースの修正を依頼 |
| ステップ8 | 実装が意図通りか（変更ファイル一覧・diff） | コミット → 次フェーズへ | ステップ3に戻り指示を修正 |
| 最終（フェーズ2・3完了後） | テスト結果・動作が意図通りか | push → PR作成 | ステップ8に戻り追加指示 |

## 3. テスト設計

### 3.1 テストケース生成方針

- Claudeが実装コードと同時にテストケース一覧を生成する
- テストケースはユーザーが確認・承認してからテストコードを生成する
- テストケースの粒度：機能単位で「正常系・異常系・境界値」を網羅する
- 既存コード修正時は既存テストケースとの差分（追加・変更・削除）を提示する

### 3.2 テストコード生成方針

- ユーザー承認後にClaudeがテストコードを生成する
- テストコマンドはClaudeがリポジトリの構成から自動判断する
- 新規作成時はテストコードを新規生成、既存修正時は既存テストを確認した上で修正・追加する

### 3.3 テスト実行の段階的実装計画

要件書（requirements (8).md）に定義された3層構成を以下の順序で実装する。

| フェーズ | 内容 | 結果保存形式 |
|---|---|---|
| フェーズ1 | 単体テスト（依存なし、高速） | Markdown |
| フェーズ2 | 結合テスト（DB・API接続が必要） | Markdown |
| フェーズ3 | E2Eテスト（Playwright、ブラウザ操作） | Markdown + スクリーンショット |
| 将来対応 | Excel形式のテスト報告書自動生成 | Excel |

### 3.4 自動修正ループ

- 最大リトライ回数: 3回
- 失敗時のフィードバック内容: 失敗したテスト名・エラーメッセージ・標準出力
- E2Eテスト失敗時はスクリーンショットもフィードバックに含める
- リトライ上限到達時はユーザーに通知して手動対応を促す

### 3.5 テスト結果の保存

- 保存場所: `/workspace/repo/test-reports/` 配下（リポジトリ内）
- ファイル名: `test-report-{日時}-{テスト種別}.md`
- 保存内容: テストケース一覧・各テストの成否・エラーメッセージ・実行日時・リトライ回数
- 将来対応: Excel形式（単体テスト報告書・結合テスト報告書・E2Eテスト報告書）

## 4. 実装確認画面（ステップ8）

- 変更ファイル一覧を右ペインに表示する
- ファイルをクリックすると diff を別画面（モーダルまたは別ページ）で表示する
- 「承認」「差し戻し」ボタンを表示する
- 差し戻し時はコメント入力欄を設け、修正の意図をClaudeに伝えられるようにする
