# Karakuri — AI駆動開発プラットフォーム 要件定義書

## 1. プロジェクト概要

### 1.1 背景

GitHub Actions + Claude Code による AI駆動開発を行っているが、以下の課題がある。

- リポジトリ上の操作のみで、ローカルでのビルド・テスト等の動作確認ができない
- 修正のたびにmasterから新しいブランチが作成され、同一ブランチでの継続的な修正ができない
- すべてのコミットがClaude名義になり、開発担当者の名前でコミットできない

### 1.2 目的

GitHub Actionsの長所（指示を出したら放置して待てる非同期性）を維持しつつ、上記の課題を解決する独自Webアプリケーションを構築する。これにより、複数タスクの同時並行実行が可能となり、開発効率の大幅な向上を目指す。

### 1.3 利用者

当面は個人（1名）での利用を想定する。

## 2. システム構成

### 2.1 アーキテクチャ概要

```
[GitHub]
  ├→ Webhook通知 ─→ [バックエンドサーバー（Python / FastAPI）]
  └← GitHub API ──┘        ↑ HTTP/WebSocket
                      [ブラウザ: Web UI]
                            
[バックエンドサーバー]
      ↓ Docker SDK（docker-py）
[タスクコンテナA] [タスクコンテナB] [タスクコンテナC] ...
   ↓                ↓                ↓
[各コンテナ内]
  - gitリポジトリ（clone）
  - Claude Code CLI
  - ビルド・テスト環境
  - ヘッドレスブラウザ（Playwright + Chromium）
  - GitHub へのpush
```

### 2.2 技術スタック

| レイヤー | 技術 |
|---|---|
| フロントエンド | React（Vite） |
| バックエンド | Python（FastAPI） |
| タスク実行環境 | Docker コンテナ |
| コンテナ管理 | docker-py（Docker公式Python SDK） |
| 非同期処理 | asyncio + Celery（タスクキュー） |
| データ永続化 | PostgreSQL |
| リアルタイム通信 | FastAPI WebSocket |
| AI連携 | Anthropic Python SDK |
| GitHub連携 | GitHub REST API / Webhook |
| 認証 | GitHub OAuth 2.0 |
| 外部公開 | Cloudflare Tunnel |
| ファイル変換 | LibreOffice（headless）、openpyxl、python-docx、pymupdf |
| ドキュメント生成 | openpyxl（Excel）、python-docx（Word）、テンプレートエンジン |

### 2.3 技術スタック選定理由

バックエンドにPython / FastAPIを採用する理由は以下の通り。

- docker-py（Docker公式SDK）が充実しており、コンテナ管理が扱いやすい
- Anthropic Python SDKにより、将来的にClaude Code CLIを介さない直接API連携が可能
- AI関連ライブラリ（LangChain、LlamaIndex等）との統合が容易で、RAGやコードレビュー自動化等の機能拡張に対応しやすい
- FastAPIの非同期処理により、複数タスクの並行管理とWebSocketストリーミングを効率的に実装できる

## 3. コンテナのライフサイクル

### 3.1 基本方針

コンテナのライフサイクルをIssue単位で管理する。GitHub Actionsのように毎回破棄するのではなく、Issueがクローズされるまでコンテナを維持し、同一ブランチ上での継続的な修正を可能にする。

### 3.2 ライフサイクルフロー

```
1. Issue対応開始（Web UIからの手動作成 or GitHub Webhookからの自動検知）
   └→ Dockerコンテナ作成
   └→ リポジトリをclone
   └→ 指定ブランチをcheckout（または新規作成）
   └→ git user.name / user.email を担当者に設定

2. 開発作業（繰り返し）
   └→ Web UIから指示を入力 + ファイルアップロード（任意）
   └→ アップロードファイルの自動変換（Excel/Word → HTML等）
   └→ プロンプト生成AIが最適なプロンプトに変換
   └→ 【必須】ユーザーがプロンプトを確認・編集 → 確定
   └→ 以降すべて自動:
       └→ コンテナ内でClaude Code CLIが実行
       └→ 実行結果・ログをWeb UIにストリーミング
       └→ 自動テスト実行（→ 失敗時は自動修正ループ）
       └→ テスト成功後、自動でコミット・push

3. 完了
   └→ PR作成（Web UIから操作）
   └→ マージ
   └→ Issueクローズ（GitHub上のIssueも自動クローズ）
   └→ コンテナ破棄
```

### 3.3 Claude Codeのコンテキスト維持

コンテナが維持される限り、Claude Codeの会話コンテキストも保持される。これにより、前回の指示内容や生成したコードを把握した状態で次の指示を受けられる。

## 4. GitHub Issue連携

### 4.1 概要

Web UIとGitHub Issueを双方向で連携する。Web UIからGitHub Issueを作成する機能と、GitHub上に作成されたIssueを自動検知して実行する機能の2つを提供する。

### 4.2 機能A：Web UIからGitHub Issueを作成

Web UIでタスクを作成する際に、同時にGitHub Issueを作成できる。

- Issue タイトル・本文をWeb UIで入力
- ラベル（例: `ai-task`）を自動付与
- 担当者（Assignee）を設定
- 作成後、そのままタスク（コンテナ）を起動して作業開始

### 4.3 機能B：GitHub Issueの自動検知・実行

GitHub上でIssueが作成（またはラベル付与）された際に、Webhookで本アプリに通知し、自動的にタスクを起動する。

```
GitHub上でIssueが作成される
  ↓
GitHub Webhook → 本アプリに通知
  ↓
セキュリティチェック（4.5参照）
  ↓
タスク（コンテナ）を起動して作業開始
```

### 4.4 Webhook設定

| 項目 | 値 |
|---|---|
| イベント | issues（opened, labeled） |
| 通知先 | 本アプリのWebhookエンドポイント |
| 認証 | Secret tokenによる署名検証（HMAC-SHA256） |
| 外部公開方法 | Cloudflare Tunnel（無料・固定ドメイン） |

### 4.5 セキュリティ（多層防御）

パブリックリポジトリでは誰でもIssueを作成できるため、第三者の指示によってコードが変更される危険がある。本アプリはアプリ自身の認証情報でpushを行うため、GitHubのリポジトリ権限だけではこのリスクを防げない。以下の多層防御を実施する。

```
GitHub Webhook受信
  ↓
① Webhook署名検証（Secret tokenによるHMAC-SHA256検証）
  → 偽装リクエストを排除
  ↓
② ラベルチェック（必須ラベルが付与されているか）
  → パブリックリポジトリではWrite権限がないユーザーはラベルを付けられない
  ↓
③ 実行許可ユーザーチェック（Issue作成者が許可リストに含まれるか）
  → 許可リストにないユーザーのIssueは無視
  ↓
④ 実行モード判定（auto / approval）
  → approvalモードの場合、Web UIに承認リクエストを表示
  ↓
⑤ タスク実行
```

### 4.6 リポジトリごとの実行ポリシー設定

リポジトリの公開状態に応じて、安全な実行ポリシーを設定する。

```yaml
# リポジトリごとの設定例

# プライベートリポジトリ（信頼済みメンバーのみアクセス可能）
private_repo_example:
  auto_execution:
    mode: "auto"                    # 自動実行
    required_label: "ai-task"
    allowed_users: []               # 空 = リポジトリアクセス可能な全員を許可

# パブリックリポジトリ（第三者がIssue作成可能）
public_repo_example:
  auto_execution:
    mode: "approval"                # 承認待ち（推奨）
    required_label: "ai-task"
    allowed_users:                  # 自動実行を許可するユーザーを明示
      - "my-github-username"
    reject_unknown_users: true      # 許可リスト外のユーザーは無視
```

### 4.7 パブリックリポジトリにおける推奨設定

パブリックリポジトリでは以下を推奨する。

- 実行モードは **approval**（承認待ち）に設定する
- 許可ユーザーリストを明示的に設定する
- 許可リスト外のユーザーが作成したIssueは **無視**（通知もしない）する
- 許可ユーザーのIssueであっても、承認画面でIssue内容を確認してから実行する

### 4.8 Issue ステータスの同期

本アプリのタスクステータスとGitHub Issueのステータスを同期する。

| 本アプリのイベント | GitHub Issueへの反映 |
|---|---|
| タスク開始 | Issueにコメント「作業を開始しました」 |
| テスト成功・push完了 | Issueにコメント「コミット完了（コミットハッシュ）」 |
| テスト失敗（リトライ上限） | Issueにコメント「自動修正に失敗しました。確認をお願いします」 |
| PR作成 | IssueにPRリンクをコメント |
| タスク完了・Issueクローズ | GitHub Issueをcloseする |

## 5. プロンプト変換・確認

### 5.1 基本方針

ユーザーの簡潔な指示をそのままClaude Codeに渡すと精度が低いため、プロンプト生成AIが最適なプロンプトに変換し、ユーザーが確認・編集してから実行する。確認ステップは必須とし、確定後はテスト・コミット・pushまですべて自動で実行される。

### 5.2 プロンプト変換フロー

```
ユーザーが簡潔な指示を入力
  例:「ログイン画面にバリデーション追加して」
  + ファイルアップロード（仕様書、画面キャプチャ等）
  ↓
プロンプト生成AI（Claude API）が以下を考慮して最適なプロンプトに変換:
  - リポジトリの構成・既存コード
  - アップロードされたファイルの内容
  - 対象ファイル・関数の特定
  - 具体的な実装手順・制約条件の明確化
  - テスト観点の提示
  ↓
変換されたプロンプトをWeb UIに表示
  ↓
【必須】ユーザーが確認・編集
  - プロンプトの内容を確認
  - 不足があれば手動で修正・追記
  - 参照すべきファイルの指定を追加・変更
  ↓
確定ボタンを押下
  ↓
以降すべて自動実行（Claude Code → テスト → 自動修正ループ → コミット → push）
```

### 5.3 プロンプト変換時のコンテキスト

プロンプト生成AIには以下の情報を入力として渡す。

- ユーザーの元の指示テキスト
- アップロードされたファイル（変換済みHTML、画像等）
- リポジトリのディレクトリ構造
- 直近の変更履歴（git log）
- 過去の指示・プロンプト履歴（同一タスク内）
- リポジトリのREADMEや設定ファイル

### 5.4 プロンプト確認画面

Web UIにプロンプト確認・編集画面を設ける。

- 変換前の元の指示（参照用）
- 変換後のプロンプト（編集可能なテキストエリア）
- アップロードファイルの一覧とプレビュー
- 参照ファイル一覧（プロンプト内で言及されているソースファイル）
- 確定ボタン / 再変換ボタン（指示を修正して再度変換を試みる）

## 6. ファイルアップロード・変換

### 6.1 基本方針

仕様書（Excel、Word等）をアップロードし、Claude Codeが作業中にいつでも参照できるようにする。Excel/Wordは表のレイアウト、セル結合、埋め込み画像といった重要な情報を保持するため、LibreOfficeによるHTML変換をベースとしたハイブリッド方式で変換する。

### 6.2 対応ファイル形式と変換方式

| ファイル種別 | 拡張子 | 変換方式 |
|---|---|---|
| Excel | .xlsx, .xls | LibreOffice → HTML変換 |
| Word | .docx, .doc | LibreOffice → HTML変換 |
| PowerPoint | .pptx, .ppt | LibreOffice → HTML変換（スライド単位） |
| PDF | .pdf | テキスト抽出（pymupdf） + 画像ページはOCR |
| 画像 | .png, .jpg, .gif | そのまま保持（Claude Codeがビジョンで参照） |
| テキスト系 | .md, .txt, .csv, .json | そのまま保持 |
| ソースコード | .py, .js, .ts 等 | そのまま保持 |

### 6.3 HTML変換パイプライン（Excel/Word/PowerPoint）

```
ファイルアップロード
  ↓
① LibreOffice（headless）でHTML変換
   → テキスト・表・セル結合・文字装飾がHTML構造として保持される
  ↓
② 埋め込み画像を抽出して個別ファイル化
   → 画面キャプチャや図はClaude Codeがビジョン機能で参照
  ↓
③ HTMLに画像パスを埋め込み直し
   → 1つのHTMLファイルで仕様書全体を閲覧可能にする
  ↓
④ 変換結果をWeb UIでプレビュー表示
   → ユーザーが正しく変換されているか確認
  ↓
⑤ 原本 + 変換済みHTML + 抽出画像 をコンテナに配置
```

### 6.4 コンテナ内のファイル配置

```
/workspace/docs/
  ├─ originals/              # アップロード原本
  │    ├─ 画面仕様書.xlsx
  │    └─ API設計書.docx
  ├─ html/                   # HTML変換済みファイル（テキスト+表+画像参照）
  │    ├─ 画面仕様書.html
  │    └─ API設計書.html
  ├─ images/                 # 埋め込み画像（抽出済み）
  │    ├─ 画面仕様書_image001.png
  │    ├─ 画面仕様書_image002.png
  │    └─ API設計書_image001.png
  └─ text/                   # 補助テキスト（検索・プロンプト補助用）
       ├─ 画面仕様書.md
       └─ API設計書.md
```

Claude Codeへのプロンプトには「/workspace/docs/html/ にある仕様書を参照してください」と含めることで、実装中にいつでも仕様を確認できる。

### 6.5 ファイルの利用タイミング

アップロードされたファイルは以下の複数のタイミングで利用される。

| タイミング | 利用形式 | 目的 |
|---|---|---|
| プロンプト変換時 | HTML + 画像 | 指示を具体的なプロンプトに変換するための仕様理解 |
| Claude Code実行時 | コンテナ内のHTML + 画像 | 実装中に仕様を随時確認 |
| 自動修正ループ時 | コンテナ内のHTML + 画像 | テスト失敗時の修正方針判断に仕様を参照 |

### 6.6 ファイルの追加アップロード

タスク作業中にも追加でファイルをアップロードできる。新しい仕様書の追加、修正された仕様書の差し替え、バグのスクリーンショットの添付等に対応する。追加されたファイルは同様の変換パイプラインを経てコンテナに配置される。

### 6.7 変換プレビューと手動修正

アップロード後、Web UIで変換結果のプレビューを表示する。変換精度に問題がある場合（レイアウト崩れ、文字化け等）、ユーザーが変換済みテキストを手動で修正できる編集機能を提供する。

### 6.8 変換に必要なソフトウェア

| ソフトウェア | 用途 | 導入先 |
|---|---|---|
| LibreOffice（headless） | Excel/Word/PowerPoint → HTML変換 | バックエンドサーバー |
| openpyxl | Excel補助処理（シート名取得等） | バックエンドサーバー |
| python-docx | Word補助処理（構造解析等） | バックエンドサーバー |
| pymupdf (fitz) | PDFテキスト抽出 | バックエンドサーバー |
| Tesseract OCR | スキャンPDFのテキスト抽出 | バックエンドサーバー |

## 7. テスト戦略

### 7.1 基本方針

テストは単体テスト・結合テスト・E2Eテストの3層構成とする。Claude Codeに実装コードとテストコードを同時に生成させ、すべてのテストをコンテナ環境内で自動実行する。テスト失敗時はClaude Codeにフィードバックし、自動修正を試みる。

### 7.2 テストの3層構成

| 層 | 対象 | 実行速度 | 環境依存 |
|---|---|---|---|
| 単体テスト | 関数・メソッド単位 | 高速 | なし |
| 結合テスト | 複数モジュール連携、DB操作、API呼び出し | 中程度 | DB・外部サービスが必要 |
| E2Eテスト | ブラウザによる画面操作全体 | 低速 | アプリ起動 + ブラウザが必要 |

### 7.3 テストコード自動生成

Claude Codeに実装コードとテストコードを同時に生成させる。プロンプト変換（セクション5）の段階で「テストも書くこと」を含めたプロンプトを生成する。

```
プロンプトに含める指示例:
  - 実装に対応する単体テストを作成すること
  - 新規API・DB操作がある場合は結合テストも作成すること
  - 画面変更がある場合はE2Eテストも作成・更新すること
  - 仕様書（/workspace/docs/html/）に基づいてテスト観点を網羅すること
```

### 7.4 テスト実行順序

テストは依存関係と実行速度を考慮し、段階的に実行する。前段が失敗した場合、後段は実行しない。

```
Claude Codeがコード修正完了
  ↓
① リント（静的解析）
  ↓ 成功
② 単体テスト（依存なし、高速）
  ↓ 成功
③ ビルド確認
  ↓ 成功
④ テスト用サービス群を起動（DB、Redis等）
  ↓
⑤ 結合テスト（DB接続、API呼び出し等）
  ↓ 成功
⑥ テスト対象アプリを起動
  ↓
⑦ E2Eテスト（Playwrightによるブラウザテスト）
  ↓ 成功
⑧ 全テスト成功 → 自動コミット → 自動push → 完了通知
```

### 7.5 自動修正ループ

```
テスト実行 ←──────────────────────┐
  ↓                                │
全テスト成功？                      │
  ├─ Yes → 自動コミット             │
  │        → 自動push               │
  │        → 完了通知                │
  └─ No  → 失敗情報を収集           │
           ├─ エラーメッセージ       │
           ├─ 失敗したテスト名       │
           ├─ スクリーンショット     │
           │  （E2Eテスト失敗時）    │
           └→ Claude Codeに渡す     │
              ↓                     │
             Claude Codeが自動修正   │
              ↓                     │
             リトライ回数 ≤ 3？      │
              ├─ Yes → 再テスト ────┘
              └─ No  → 失敗通知
                       （人間の判断を仰ぐ）
```

### 7.6 リトライ制御

| 項目 | 値 |
|---|---|
| 自動修正リトライ上限 | 3回 |
| リトライ上限到達時 | 失敗通知を送り、人間の判断を仰ぐ |
| テスト成功時 | 自動でコミット・pushまで実行する |

### 7.7 テスト結果のフィードバック形式

テスト失敗時、以下の情報をClaude Codeに渡して自動修正を促す。

- 失敗したテストの種別（単体 / 結合 / E2E）
- 失敗したテストコマンド名
- 標準出力・標準エラー出力（テストのエラーメッセージ）
- 失敗したテストファイル・テスト名（パース可能な場合）
- E2Eテスト失敗時のスクリーンショット（Playwright自動撮影）
- 現在のリトライ回数（「残りN回で解決してください」）

## 8. テスト環境（Docker Compose）

### 8.1 基本方針

結合テスト・E2Eテストに必要な外部サービス（DB、キャッシュ、アプリサーバー等）はDocker Composeでタスクごとに起動する。これにより、すべてのテストをコンテナ環境内で完結させ、非同期・自動実行の流れを維持する。

### 8.2 Docker Compose構成

```yaml
services:
  workspace:              # Claude Codeが動く作業環境
    image: karakuri-workspace
    volumes:
      - repo:/workspace
      - test-data:/workspace/test-data

  db:                     # テスト用データベース
    image: postgres:16
    environment:
      POSTGRES_DB: test_db
      POSTGRES_PASSWORD: test
    volumes:
      - ./test-data/seed.sql:/docker-entrypoint-initdb.d/seed.sql

  redis:                  # キャッシュ等（必要な場合）
    image: redis:7

  app:                    # テスト対象のWebアプリケーション
    build: ./repo
    command: "npm run dev"
    ports:
      - "3000:3000"
    depends_on:
      - db
      - redis

  # Playwrightはworkspaceコンテナ内から実行
  # → http://app:3000 にアクセスしてE2Eテスト
```

### 8.3 リポジトリごとのテスト環境設定

リポジトリごとに必要なサービスとテストコマンドを設定する。

```yaml
# リポジトリごとの設定例
test_services:
  - name: "postgres"
    image: "postgres:16"
    env:
      POSTGRES_DB: "test_db"
      POSTGRES_PASSWORD: "test"
    port: 5432
    init_script: "seed.sql"       # 共有データから初期化スクリプトを指定

  - name: "redis"
    image: "redis:7"
    port: 6379

app_server:
  build_command: "npm run build"
  start_command: "npm run dev"
  port: 3000
  health_check: "http://localhost:3000/health"
  depends_on: ["postgres", "redis"]

test_commands:
  - name: "リント"
    command: "npm run lint"
    required: true
    type: "lint"

  - name: "単体テスト"
    command: "npm test -- --unit"
    required: true
    type: "unit"

  - name: "ビルド確認"
    command: "npm run build"
    required: true
    type: "build"

  - name: "結合テスト"
    command: "npm test -- --integration"
    required: true
    type: "integration"
    depends_on: ["postgres", "redis"]

  - name: "E2Eテスト"
    command: "npx playwright test"
    required: false
    type: "e2e"
    depends_on: ["app"]
```

### 8.4 テストサービスのライフサイクル

```
タスク開始
  ↓
workspaceコンテナ起動（常時）
  ↓
テスト実行時:
  ├─ 単体テスト → 追加サービス不要、そのまま実行
  ├─ 結合テスト → DB・Redis等を起動 → テスト実行 → サービス停止
  └─ E2Eテスト → DB・Redis・アプリ起動 → テスト実行 → サービス停止
  ↓
テスト完了後、テスト用サービスは停止（リソース解放）
workspaceコンテナはタスク完了まで維持
```

テスト用サービスはテスト実行時のみ起動し、終了後は停止する。これにより常時リソースを消費することを避ける。

## 9. テストデータ管理

### 9.1 基本方針

テストに必要なデータファイル（DBダンプ、マスターデータ、テスト用画像等）のうち、リポジトリに含まれないものを管理する仕組みを提供する。データの性質に応じて3つの方式を組み合わせる。

### 9.2 データの種類と管理方式

| データの性質 | 方式 | 説明 |
|---|---|---|
| 毎回同じテストデータ | 共有ボリューム | サーバー上に永続化し、コンテナ起動時に自動マウント |
| タスクごとに異なるデータ | ファイルアップロード | Web UIから都度アップロード |
| DB初期データ | 初期化スクリプト | Docker Composeの初期化機能で自動投入 |
| 秘匿情報（APIキー等） | 設定画面で管理 | 環境変数としてコンテナに注入 |

### 9.3 共有ボリューム

プロジェクトで繰り返し使うテストデータは、サーバー上の共有ボリュームに永続化する。コンテナ起動時に自動マウントされるため、毎回アップロードする必要がない。

```
サーバー上の永続ストレージ
  /data/{リポジトリ名}/test-data/
    ├─ seed.sql                  # DB初期データ
    ├─ master_data.csv           # マスターデータ
    └─ fixtures/
         ├─ sample_images/       # テスト用画像
         └─ mock_responses/      # 外部APIモックデータ

  ↓ Docker起動時に自動マウント

コンテナ内
  /workspace/test-data/          # 読み書き可能
```

### 9.4 DB初期データの投入

Docker Composeの初期化機能を利用し、テスト用DBに自動的にデータを投入する。

```yaml
services:
  db:
    image: postgres:16
    volumes:
      # コンテナ起動時にseed.sqlが自動実行される
      - /data/my-project/test-data/seed.sql:/docker-entrypoint-initdb.d/seed.sql
```

### 9.5 環境変数・秘匿情報の管理

APIキー、外部サービスの認証情報等は、設定画面で管理し、環境変数としてコンテナに注入する。

```yaml
# リポジトリごとの設定例
environment_variables:
  - name: "DATABASE_URL"
    value: "postgresql://test:test@db:5432/test_db"
    secret: false

  - name: "EXTERNAL_API_KEY"
    value: "sk-xxxxx"
    secret: true         # 暗号化して保存、Web UIでマスク表示
```

### 9.6 コンテナ内のデータ配置（全体像）

```
/workspace/
  ├─ repo/                    # gitリポジトリ（clone済み）
  ├─ docs/                    # 仕様書（セクション6で定義）
  │    ├─ originals/
  │    ├─ html/
  │    ├─ images/
  │    └─ text/
  └─ test-data/               # テストデータ
       ├─ seed.sql
       ├─ master_data.csv
       └─ fixtures/
            ├─ sample_images/
            └─ mock_responses/
```

### 9.7 Web UIでのテストデータ管理

設定画面にテストデータ管理機能を設ける。

- 共有データファイルのアップロード・一覧・削除・更新
- DB初期化スクリプトの指定
- 環境変数（秘匿情報含む）の設定・編集
- マウントパスの設定
- テストデータのバージョン管理（更新日時の記録）

## 10. E2Eテスト（Playwright）

### 10.1 基本方針

E2EテストにはPlaywrightを採用する。テスト対象のWebアプリケーションをDocker Compose内で起動し、Playwrightがヘッドレスブラウザでアクセスしてテストを実行する。

### 10.2 Playwright採用理由

| 観点 | Playwright | 他の選択肢との比較 |
|---|---|---|
| ヘッドレス実行 | 標準対応 | Cypressも対応するがDocker相性はPlaywrightが優れる |
| Docker対応 | 公式イメージあり | 環境構築が容易 |
| 複数ブラウザ | Chromium / Firefox / WebKit | Cypressはchromiumのみ |
| 速度 | 高速（並列実行対応） | Seleniumより大幅に高速 |
| スクリーンショット | テスト失敗時に自動撮影 | 自動修正ループのフィードバックに活用 |
| テスト自動生成 | codegen機能あり | ブラウザ操作を記録してコード生成可能 |

### 10.3 E2Eテストコードの生成方針

E2Eテストコードの生成はClaude Codeが仕様書ベースで自動生成する。テスト失敗時はスクリーンショットをフィードバックして修正精度を向上させる。

```
Claude Codeが実装コード生成時に、同時にE2Eテストコードも生成
  ↓
  （仕様書の画面キャプチャ・HTML変換を参照して生成）
  ↓
E2Eテスト実行
  ↓
失敗した場合:
  ├─ Playwrightが失敗時のスクリーンショットを自動撮影
  ├─ エラーメッセージ + スクリーンショットをClaude Codeに渡す
  │    → 「この画面でこのエラーが出ています。
  │       テストまたは実装を修正してください」
  └─ Claude Codeが実装コードまたはテストコード自体を修正
      ↓
     再テスト（自動修正ループ内で処理）
```

### 10.4 E2Eテスト実行環境

```yaml
# workspaceコンテナにPlaywright環境を含める
workspace:
  image: karakuri-workspace  # Playwrightの公式Dockerイメージをベース
  # 含まれるもの:
  #   - Playwright
  #   - Chromium（ヘッドレス）
  #   - Firefox（ヘッドレス）
  #   - WebKit（ヘッドレス）

# テスト対象アプリ
app:
  build: ./repo
  command: "npm run dev"
  ports:
    - "3000:3000"
```

Playwrightはworkspaceコンテナ内から `http://app:3000` にアクセスしてテストを実行する。

### 10.5 スクリーンショットの管理

Playwrightが撮影したスクリーンショットは以下の用途で利用する。

| 用途 | 説明 |
|---|---|
| 自動修正フィードバック | テスト失敗時のスクリーンショットをClaude Codeに渡し、修正精度を向上 |
| テスト結果の可視化 | Web UIでスクリーンショットを閲覧し、テスト結果を視覚的に確認 |
| 回帰テストのベースライン | 正常時のスクリーンショットを保持し、ビジュアルリグレッション検知に活用（将来拡張） |

```
/workspace/test-results/
  ├─ screenshots/
  │    ├─ login-page-success.png
  │    ├─ login-page-failure.png    # テスト失敗時の自動撮影
  │    └─ dashboard-loaded.png
  └─ reports/
       └─ playwright-report.html    # Playwrightのテストレポート
```

### 10.6 Web UIでの表示

タスク作業画面にテスト関連の以下の情報を表示する。

- 現在のテスト実行状態（実行中 / 成功 / 失敗 / リトライ中）
- テスト実行フェーズ（リント → 単体 → ビルド → 結合 → E2E）の進捗表示
- 各テストコマンドの成否と出力ログ
- E2Eテスト失敗時のスクリーンショット表示
- Playwrightテストレポートの閲覧
- 自動修正ループの進捗（リトライ回数、現在の状況）
- テストサービス（DB、Redis等）の起動状態
- 手動テスト実行ボタン（任意のタイミングで特定のテストを実行可能）

## 11. 仕様データベース

### 11.1 基本方針

開発対象の仕様情報を構造化データとして蓄積・管理する。仕様データベースはプロジェクト開始時に初期構築し、Issue対応で修正が入るたびに差分更新する。常に最新の仕様データが維持されることで、設計書・テスト仕様書等の納品ドキュメントを高精度に自動生成できる。

### 11.2 共通データ構造

仕様データベースは「機能の最小単位」を共通構造とする。どのようなプロジェクト（Webアプリ、バッチ処理、モバイルアプリ等）でも、以下の構造に分解して管理する。プロジェクトによって使わない層は空のままでよい。

```
プロジェクト（Project）
  └─ 機能（Feature）
       ├─ 画面（Screen）
       │    └─ 画面要素（Element）
       ├─ API（Endpoint）
       │    ├─ リクエストパラメータ（Parameter）
       │    └─ レスポンス（Response）
       ├─ データ（Table / Entity）
       │    └─ カラム（Column / Field）
       └─ ビジネスロジック（Rule）
```

### 11.3 各層の項目定義

**プロジェクト（Project）**

| 項目 | 説明 |
|---|---|
| プロジェクト名 | 対象プロジェクトの名称 |
| 概要 | プロジェクトの目的・概要 |
| 技術スタック | 使用言語、フレームワーク、DB等 |
| 非機能要件 | 性能、セキュリティ、可用性等の要件 |

**機能（Feature）**

| 項目 | 説明 |
|---|---|
| 機能ID | 一意の識別子（例: F001） |
| 機能名 | 機能の名称 |
| 概要 | 機能の目的・動作概要 |
| ステータス | 未着手 / 開発中 / 完了 / 変更中 |
| 関連Issue | 紐づくGitHub Issue番号 |

**画面（Screen）**

| 項目 | 説明 |
|---|---|
| 画面ID | 一意の識別子（例: S001） |
| 画面名 | 画面の名称 |
| URL / パス | 画面のルーティングパス |
| 画面遷移 | 遷移元・遷移先の画面ID |
| 認証要否 | ログイン必須かどうか |

**画面要素（Element）**

| 項目 | 説明 |
|---|---|
| 要素名 | 表示名またはラベル |
| 種別 | テキスト入力 / ボタン / テーブル / ドロップダウン等 |
| バリデーション | 入力チェック条件 |
| アクション | 要素操作時の動作（API呼び出し、画面遷移等） |
| 表示条件 | 表示/非表示の条件 |

**API（Endpoint）**

| 項目 | 説明 |
|---|---|
| API ID | 一意の識別子（例: A001） |
| メソッド | GET / POST / PUT / DELETE |
| パス | エンドポイントのURL |
| 認証 | 認証方式（JWT、APIキー等） |
| リクエスト | パラメータ一覧（名前、型、必須、説明） |
| レスポンス | 正常時・異常時のレスポンス定義 |
| エラーコード | エラー時のステータスコードとメッセージ |

**データ（Table / Entity）**

| 項目 | 説明 |
|---|---|
| テーブル名 | テーブルまたはエンティティの名称 |
| カラム一覧 | 名前、型、制約（NOT NULL、UNIQUE等） |
| 主キー | 主キーカラム |
| インデックス | インデックス定義 |
| リレーション | 他テーブルとの関連（外部キー等） |

**ビジネスロジック（Rule）**

| 項目 | 説明 |
|---|---|
| ルールID | 一意の識別子（例: R001） |
| ルール名 | ビジネスルールの名称 |
| 条件 | ルールが適用される条件 |
| 処理内容 | 条件を満たした場合の処理 |
| 例外処理 | 異常時の処理 |

### 11.4 仕様データベースのライフサイクル

**プロジェクト開始時（初期構築）**

既存の仕様書がある場合とゼロから作る場合の両方に対応する。

```
パターンA: 既存仕様書がある場合
  仕様書（Excel/Word/PDF）をアップロード
    ↓
  AIが仕様データベースの初期データを自動抽出
    ↓
  ユーザーが確認・修正
    ↓
  仕様データベース（ベースライン）完成

パターンB: ゼロから作る場合
  Web UIで機能一覧を入力（または指示で生成させる）
    ↓
  各機能の詳細をAIが対話的に生成
    ↓
  ユーザーが確認・修正
    ↓
  仕様データベース（ベースライン）完成
```

**通常の開発サイクル（差分更新）**

```
Issue対応の指示入力
  ↓
プロンプト変換時にAIが仕様データベースの変更点を自動検出
  例:「ログイン画面にバリデーション追加」
    → 画面仕様S001のelementsにvalidation項目を追加提案
  ↓
プロンプト確認画面で仕様変更も一緒に確認・編集
  ↓
確定 → 仕様データベース更新 + Claude Code実行
  ↓
実装完了後、コードと仕様データベースの整合性チェック（補完）
```

### 11.5 プロンプト確認画面との統合

セクション5のプロンプト確認画面を拡張し、仕様変更も同時に確認できるようにする。

```
プロンプト確認画面（拡張版）:
  ┌─────────────────────────────────────────┐
  │ [1] 元の指示                             │
  │ 「ログイン画面にバリデーション追加」       │
  ├─────────────────────────────────────────┤
  │ [2] 変換後プロンプト          [編集可能]  │
  │ （Claude Codeへの詳細な指示）             │
  ├─────────────────────────────────────────┤
  │ [3] 仕様データベース変更点     [編集可能]  │
  │  変更: S001 ログイン画面                  │
  │   追加: email入力欄 - バリデーション       │
  │     「メール形式チェック」                 │
  │   追加: password入力欄 - バリデーション    │
  │     「8文字以上、英数字混在」              │
  ├─────────────────────────────────────────┤
  │ [再変換]  [確定して実行]                   │
  └─────────────────────────────────────────┘
```

### 11.6 仕様データベースの保存先

仕様データベースはバックエンドのデータベース（PostgreSQL）に保存する。リポジトリごとに独立したスキーマを持つ。

### 11.7 Web UIでの閲覧・編集（優先度: 低）

仕様データベースの内容をWeb UIで直接閲覧・編集できる画面を設ける。ただし初期バージョンでは必須としない。

- 機能一覧のツリー表示（Feature → Screen → Element の階層表示）
- 各項目の詳細表示・編集フォーム
- 変更履歴の表示（いつ、どのIssueで変更されたか）
- 検索機能（機能名、API名、テーブル名等で検索）

## 12. ドキュメント自動生成

### 12.1 基本方針

仕様データベースとテスト実行結果を元に、指定されたテンプレートに差し込んでExcel/Word形式の納品ドキュメントを自動生成する。生成タイミングは工程完了時にユーザーが手動で指定する。

### 12.2 対応ドキュメント一覧

| フェーズ | ドキュメント | 形式 | データソース |
|---|---|---|---|
| 要件定義 | 要件定義書 | Word | 仕様DB: プロジェクト概要 + 機能一覧 + 非機能要件 |
| 設計 | 基本設計書 | Word | 仕様DB: 機能仕様 + 画面遷移 + API一覧 |
| 設計 | 詳細設計書 | Word | 仕様DB: 画面要素 + API詳細 + ビジネスロジック |
| 設計 | DB設計書 | Excel | 仕様DB: テーブル定義 + カラム一覧 + リレーション |
| 設計 | 画面設計書 | Excel | 仕様DB: 画面仕様 + 要素一覧 + バリデーション |
| テスト | 単体テスト仕様書 | Excel | 仕様DB: 機能仕様からテスト観点を導出 |
| テスト | 単体テスト報告書 | Excel | テスト実行結果データ + 仕様DB |
| テスト | 結合テスト仕様書 | Excel | 仕様DB: API仕様 + データ仕様からシナリオ導出 |
| テスト | 結合テスト報告書 | Excel | テスト実行結果データ + 仕様DB |
| テスト | E2Eテスト報告書 | Excel | テスト実行結果 + スクリーンショット |

### 12.3 テンプレート方式

**テンプレートの管理**

会社・プロジェクト固有のフォーマットをテンプレートとしてアップロードし、差し込みフィールドをマッピングする。

```
① 設定画面からテンプレートファイル（Excel/Word）をアップロード
  ↓
② テンプレート内の差し込みフィールドを定義
   - 表紙: {{project_name}}, {{author}}, {{date}} 等
   - データ部: {{#features}} ... {{/features}} 等のループ構造
  ↓
③ テンプレートをリポジトリ（またはドキュメント種別）に紐付け
  ↓
④ ドキュメント生成時にテンプレート + データ → 完成品を出力
```

**テンプレートの構成例（単体テスト報告書: Excel）**

```
テンプレートファイル:
  ├─ 表紙シート
  │    プロジェクト名: {{project_name}}
  │    作成日: {{date}}
  │    作成者: {{author}}
  │    対象機能: {{feature_name}}
  │
  ├─ テスト一覧シート
  │    {{#test_results}}
  │    | No | テスト名 | 結果 | 実行日時 | 備考 |
  │    | {{index}} | {{name}} | {{result}} | {{datetime}} | {{note}} |
  │    {{/test_results}}
  │
  ├─ テスト詳細シート
  │    {{#test_details}}
  │    テスト名: {{name}}
  │    入力値: {{input}}
  │    期待値: {{expected}}
  │    実際の値: {{actual}}
  │    結果: {{result}}
  │    {{/test_details}}
  │
  └─ 総括シート
       テスト総数: {{total_count}}
       成功数: {{pass_count}}
       失敗数: {{fail_count}}
       成功率: {{pass_rate}}%
       所見: {{summary}}  ← AIが自動生成
```

### 12.4 ドキュメント生成フロー

```
Web UIで [ドキュメント生成] ボタンを押下
  ↓
生成するドキュメント種別を選択
  例: 単体テスト報告書
  ↓
テンプレートを選択（リポジトリにデフォルト設定あり）
  ↓
対象範囲を選択
  例: 機能F001〜F003 / 直近のテスト実行結果
  ↓
生成実行
  ├─ 仕様データベースからデータ取得
  ├─ テスト実行結果からデータ取得
  ├─ 文章部分（所見・総括）をAIが生成
  └─ テンプレートにデータを差し込み
  ↓
プレビュー表示（Web UI上でドキュメントの内容を確認）
  ↓
必要に応じて手動修正（AI生成の文章部分等）
  ↓
ダウンロード（Excel / Word形式）
```

### 12.5 ドキュメント生成に使用するライブラリ

| 形式 | ライブラリ | 用途 |
|---|---|---|
| Excel (.xlsx) | openpyxl | テンプレートへのデータ差し込み、書式保持 |
| Word (.docx) | python-docx | テンプレートへのデータ差し込み、書式保持 |
| 文章生成 | Anthropic Python SDK | 所見・総括等の文章部分を自動生成 |

### 12.6 AI生成テキストの品質管理

ドキュメント内の「所見」「総括」「考察」等の文章部分はAIが自動生成するが、生成後に必ずプレビューで確認し、必要に応じて手動修正できるようにする。

### 12.7 ドキュメントのバージョン管理

生成されたドキュメントは日時付きで保存し、過去に生成したドキュメントも一覧・ダウンロードできるようにする。

```
/documents/{リポジトリ名}/
  ├─ 要件定義書_20260211_v1.docx
  ├─ 単体テスト報告書_F001_20260211.xlsx
  ├─ 単体テスト報告書_F002_20260212.xlsx
  └─ 結合テスト報告書_20260215.xlsx
```

## 13. 同時実行制御

### 13.1 同時オープンIssue数の制限

サーバーリソースに基づき、同時にアクティブにできるタスク数に上限を設ける。

| 項目 | 値（目安） |
|---|---|
| 同時アクティブタスク上限 | 3〜5（サーバースペックに依存） |
| 1タスクあたりメモリ（常時） | 3〜4 GB（workspaceコンテナ + Playwright） |
| 1タスクあたりメモリ（テスト時） | 5〜8 GB（上記 + DB・Redis・アプリ等のサービス） |
| 1タスクあたりディスク | 5〜10 GB |

### 13.2 上限超過時の挙動

- 新規タスク（Web UIからの手動作成、GitHub Webhook経由の自動検知ともに）はキューに追加される
- 一定時間操作のないコンテナを自動で一時停止（`docker pause`）し、リソースを解放する
- 一時停止されたコンテナは、再開指示で即座に復帰できる

### 13.3 タイムアウト

| 条件 | アクション |
|---|---|
| 操作なし 1時間 | コンテナを一時停止（`docker pause`） |
| 操作なし 24時間 | 警告通知を表示 |
| 操作なし 7日間 | コンテナを停止（`docker stop`）。データはボリュームに保持 |
| 操作なし 30日間 | コンテナとボリュームを破棄 |

※ タイムアウト値は設定で変更可能とする。

## 16. 非機能要件

### 16.1 サーバースペック目安

同時3〜5タスク（各タスクにDocker Composeでテストサービスを含む）を前提とした場合の目安：

| リソース | 推奨値 |
|---|---|
| CPU | 8コア以上 |
| メモリ | 32 GB 以上 |
| ディスク | 200 GB 以上（SSD推奨） |

※ 各タスクがworkspaceコンテナ + テスト時にDB・Redis・アプリ等のサービスコンテナを起動する。テスト用サービスはテスト実行中のみ起動するため、常時のリソース消費はworkspaceコンテナ分のみ。

### 16.2 認証・セキュリティ

**Web UI認証（GitHub OAuth）**

Web UIへのログインにはGitHub OAuthを使用する。GitHubと密に連携するアプリであるため、GitHubアカウントで認証することで権限管理との一貫性を保つ。将来のチーム利用への拡張にも対応しやすい。

| 項目 | 値 |
|---|---|
| 認証方式 | GitHub OAuth 2.0 |
| ログインフロー | GitHubの認証画面にリダイレクト → 認証後にコールバック → セッション発行 |
| セッション管理 | サーバーサイドセッション（JWTトークン） |
| アクセス制御 | 許可するGitHubユーザーIDを設定で指定 |
| セッション有効期限 | 設定で変更可能（デフォルト: 24時間） |

**GitHub API認証**

KarakuriがGitHubのリポジトリを操作するために、GitHub Personal Access Token（PAT）またはGitHub Appの認証情報を使用する。設定画面で管理し、暗号化して保存する。

**Anthropic API / Claude Code認証**

Claude Codeの利用にはAnthropicのAPIキーまたはMax Planのアカウント認証を使用する。設定画面で管理し、暗号化して保存する。

**Webhook認証**

GitHub Webhookの受信時にはSecret tokenによる署名検証（HMAC-SHA256）を必須とする（セクション4.5で定義済み）。

**その他のセキュリティ要件**

- GitHub認証情報・APIキーは暗号化して保存する
- コンテナ間のネットワークは隔離する
- パブリックリポジトリでは実行許可ユーザーリストと承認モードを推奨する

### 16.3 可用性

- サーバー再起動時、停止中のコンテナを自動で復元できること
- タスクの状態（指示履歴、テスト結果、ステータス）はデータベースに永続化する

### 16.4 外部公開

GitHub Webhookを受信するため、Cloudflare Tunnelを使用してサーバーの一部エンドポイントを外部公開する。無料で固定ドメインが利用可能であり、個人利用に適する。

### 16.5 ログ・監視

異常の種類に応じて対応方針を定める。すべてのログはファイルに記録し、Web UIからも閲覧可能にする。

**アプリケーションエラー（APIタイムアウト、Claude Code応答なし等）**

Web UIにエラー通知を表示し、タスクを一時停止する。原因不明のまま繰り返しても解決しないため、自動リトライはしない。

**インフラ障害（ディスク枯渇、メモリ不足、コンテナクラッシュ等）**

サーバーリソース（CPU、メモリ、ディスク使用率）を定期的に監視し、閾値超過時にダッシュボードに警告を表示する。コンテナクラッシュ時は自動再起動を試み、失敗した場合はタスクを停止状態にする。

**外部サービス障害（GitHub API障害、Anthropic API障害等）**

エクスポネンシャルバックオフ付きでリトライし、一定回数（5回）失敗した場合はタスクを一時停止する。サービス復旧後に手動で再開できる。

### 16.6 バックアップ

| 対象 | 方式 | 頻度 | 保持期間 |
|---|---|---|---|
| PostgreSQL（仕様DB、タスク状態等） | pg_dumpによる自動バックアップ | 日次 | 7日間 |
| テンプレートファイル | ファイルコピー | 日次（DBバックアップと同時） | 7日間 |
| テストデータ（共有ボリューム） | ファイルコピー | 日次（DBバックアップと同時） | 7日間 |
| リポジトリのコード | バックアップ不要 | — | GitHub上にpush済み |

### 16.7 通知

初期バージョンではWeb UI上の通知とブラウザプッシュ通知に対応する。ブラウザがバックグラウンドでも通知を受け取れるようにする。

**通知対象イベント**

| イベント | 通知内容 |
|---|---|
| タスク完了（テスト成功・push完了） | 「タスク○○が完了しました」 |
| テスト失敗（リトライ上限到達） | 「タスク○○の自動修正に失敗しました」 |
| 承認待ちIssue検知 | 「新しいIssueが承認待ちです」 |
| アプリケーションエラー | 「タスク○○でエラーが発生しました」 |
| インフラ警告（リソース閾値超過） | 「ディスク使用率が90%を超えました」 |
| コンテナクラッシュ | 「タスク○○のコンテナが停止しました」 |

**通知方式**

| 方式 | 対応バージョン |
|---|---|
| Web UI上の通知（ダッシュボード・ベルアイコン） | 初期バージョン |
| ブラウザプッシュ通知 | 初期バージョン |
| Slack通知 | 将来拡張 |
| メール通知 | 将来拡張 |

## 18. 用語定義

| 用語 | 定義 |
|---|---|
| タスク | 1つのIssueに対応する作業単位。Docker Composeサービス群に対応する |
| コンテナ | タスクごとに作成される隔離されたDocker実行環境 |
| workspaceコンテナ | Claude CodeとPlaywrightが動作するメインの作業コンテナ |
| テストサービス | 結合テスト・E2Eテストのためにテスト実行時に起動されるDB・Redis等のコンテナ群 |
| 指示 | Web UIからClaude Codeに送る作業依頼。1タスク内で複数回行える |
| プロンプト変換 | ユーザーの簡潔な指示をプロンプト生成AIが最適なプロンプトに変換する処理 |
| ファイル変換 | アップロードされたExcel/Word等をHTML形式に変換し、Claude Codeが参照可能にする処理 |
| 仕様データベース | 開発対象の仕様情報を機能の最小単位で構造化して蓄積するデータベース |
| ベースライン | プロジェクト開始時に構築する仕様データベースの初期状態 |
| テンプレート | ドキュメント生成に使用する会社・プロジェクト固有のExcel/Wordフォーマット |
| 差し込みフィールド | テンプレート内のデータ挿入位置を示すプレースホルダー |
| 自動修正ループ | テスト失敗時にClaude Codeへフィードバックし再修正を試みるサイクル。最大3回 |
| 共有ボリューム | テストデータ等をサーバー上に永続化し、コンテナ起動時に自動マウントする仕組み |
| 承認待ち | Webhook経由で検知されたIssueが、人間の承認を待っている状態 |
| 許可ユーザーリスト | 自動実行を許可するGitHubユーザー名のリスト |
| ダッシュボード | アクティブなタスクの一覧と状態を表示するWeb UIのトップ画面 |
